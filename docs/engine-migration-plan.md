# План выбора движка и переноса Rogue Demo

## Статус

Решение принято: проект переносится в **Phaser 3**.

Этот документ остается как запись рассуждений и сравнения вариантов.
Практический план переноса теперь ведется в [Phaser Migration Roadmap](phaser-migration-roadmap.md).

## Зачем вообще переносить

Текущий проект уже вышел из стадии одноразового прототипа:

- игровое ядро собрано вокруг `GameCore`
- есть реестр уровней и типов врагов
- есть отдельные подсистемы для рендера, UI, аудио и анимаций

Но дальше scope резко меняется: кат-сцены, сюжетные смены правил, два героя, более длинная кампания, сохранения прогресса, больше контентных пайплайнов. Для такого проекта главное не просто "мощнее рендер", а:

- удобное авторство контента
- управление сценами и состояниями игры
- инструменты для кат-сцен и scripted sequences
- нормальная работа с UI, анимациями, триггерами и сохранениями
- внятный pipeline для роста команды и объема контента

## Что уже хорошо в текущем коде

Текущая архитектура уже частично подготовлена к миграции:

- [`GameCore`](../js/game/core/GameCore.js) держит основной игровой цикл и состояние
- [`LevelRegistry`](../js/game/level/LevelRegistry.js) уже задает расширяемую модель уровней
- [`coreRules`](../js/game/systems/coreRules.js) выделяет игровые правила в отдельную систему

Это значит, что перенос можно делать не как "переписать игру с нуля", а как последовательное отделение доменной логики от браузерной оболочки.

## Что сейчас мешает легкому переносу

Есть несколько мест, где логика еще слишком тесно связана с браузером:

- [`GameCore`](../js/game/core/GameCore.js) напрямую знает про `window`, `document`, `renderer`, `hud`, `audio`
- [`DomRenderer`](../js/game/render/DomRenderer.js) жестко завязан на HTML/CSS DOM-рендер тайлов
- [`bootstrap`](../js/game/bootstrap.js) собирает игру сразу как браузерное приложение, а не как engine-agnostic core + adapter

Вывод: перенос надо начинать не с копирования ассетов в новый движок, а с выделения "чистого" gameplay core.

## Shortlist движков

### 1. Godot 4.6.1

Это основной кандидат.

Почему подходит:

- у Godot сильный 2D-пайплайн, tilemap workflow, встроенный UI и scene-driven архитектура
- для кат-сцен и scripted sequences подходят сцены, анимации и ресурсная модель
- удобно описывать игровые сущности и сюжетные состояния как data-driven ресурсы
- нет лицензионной стоимости движка, проект MIT/open source

Почему это особенно хорошо для Rogue Demo:

- структура `уровень -> сущности -> состояния -> события` хорошо ложится на сцены и ресурсы Godot
- для двух героев, смены механик и кат-сцен удобнее иметь editor-first pipeline, чем наращивать DOM-рендер
- для инди-проекта меньше операционный риск, чем у тяжелого enterprise-стека

Ограничения:

- придется выбрать язык слоя логики: GDScript или C#
- если нужен именно web-first pipeline и максимальное сохранение JS-кода, Godot даст меньше прямого переиспользования, чем Phaser
- на Godot тоже нужно будет дисциплинированно держать gameplay отдельно от scene glue code

### 2. Unity 6.3 LTS

Это разумный вариант, если приоритетом будет C#, долгий production lifecycle и более "корпоративный" toolchain.

Почему подходит:

- сильные инструменты для cinematic sequencing и камер
- хороший production pipeline для контента, анимаций, UI и сервисов
- C# удобен, если вы хотите строить более строгую codebase и нанимать людей под распространенный стек

Минусы для этого проекта:

- существенно тяжелее по редактору, пайплайну и общему overhead
- лицензионные и ценовые условия нужно учитывать заранее
- для небольшой 2D narrative roguelike-игры это может оказаться избыточным

### 3. Phaser 3.90

Это лучший вариант, если вы хотите остаться в web-first мире и максимально сохранить JavaScript/TypeScript-мышление.

Почему подходит:

- сцены, таймлайны, тайлмапы и быстрый вход для текущего JS-проекта
- проще всего перенести часть текущей логики и быстрее получить playable vertical slice
- хорошо подходит, если игра останется браузерной и вам важен быстрый deployment

Минусы:

- editor-first pipeline заметно слабее, чем у Godot и Unity
- кат-сцены, сложные narrative pipelines и контентный production придется больше собирать руками
- чем дальше проект уходит в authored content и сложные последовательности, тем сильнее будет хотеться полноценного engine/editor workflow

### 4. Defold

Это запасной компактный вариант.

Плюсы:

- легкий, быстрый, хорошо подходит для 2D
- есть scene editor, GUI, анимации и хороший runtime footprint

Минусы:

- меньше экосистема и меньше готовых production-практик под narrative-heavy игру
- Lua-стек для вашей текущей JS-архитектуры выглядит менее естественным, чем GDScript/C# или TypeScript

## Рекомендация

### Если выбирать один движок сейчас

Рекомендую **Godot 4.6.1** как основной путь.

Причина простая: ваш будущий scope больше похож на **2D story-driven game с authored content**, чем на "еще один браузерный prototype". Для такого проекта важнее:

- сцены
- инструменты для авторинга контента
- UI
- кат-сцены
- data-driven сущности
- удобный editor workflow

По совокупности это лучший баланс между:

- серьезностью пайплайна
- скоростью разработки
- стоимостью владения
- риском для маленькой команды

### Когда вместо Godot выбирать Phaser

Выбирайте Phaser, если верны сразу оба пункта:

- игра должна остаться прежде всего браузерной
- вы хотите сохранить максимум текущего JS/TS-кода и быстро собрать новый runtime

### Когда вместо Godot выбирать Unity

Выбирайте Unity, если верны хотя бы два пункта:

- нужен C# как основной язык
- планируется расширение команды под Unity-рынок
- важен более тяжелый production/tooling pipeline
- есть планы на платформы и процессы, где Unity уже привычен команде

## Что можно сделать прямо сейчас

### 1. Зафиксировать критерии выбора

Нужно принять решение не "по ощущению", а по матрице:

- скорость первого vertical slice
- удобство кат-сцен
- удобство двух героев и смены механик
- удобство UI/HUD/меню
- удобство data-driven контента
- стоимость и операционные риски
- удобство найма и входа для новых разработчиков
- экспорт на нужные платформы

Рекомендация: оценивать каждый движок по шкале `1..5`.

### 2. Сузить shortlist до двух движков

Практически я бы оставил:

- `Godot`
- `Phaser`

`Unity` стоит держать как осознанный альтернативный путь, но не как обязательный кандидат для первого технического спайка.

### 3. Сделать не "миграцию всей игры", а технический спайк

Нужен маленький вертикальный срез:

- один уровень
- один герой
- один тип врага
- пошаговое движение
- одна короткая кат-сцена
- HUD с HP и этажом

Цель спайка: понять не "можно ли", а "насколько удобно это делать именно у нас".

### 4. Подготовить текущий код к переносу

До выбора движка полезно сделать один архитектурный шаг: выделить gameplay core, который не зависит от DOM.

Минимальный целевой контур:

- `gameplay state`
- `commands` игрока и AI
- `turn resolution`
- `combat rules`
- `level data schema`
- `event stream` вида `heroMoved`, `enemyDied`, `cutsceneStarted`

### 5. Принять решение после короткого спайка, а не до него

Нормальный срок на выбор:

- `2-3 дня` на Godot spike
- `1-2 дня` на Phaser spike

После этого решение обычно становится очевидным.

## Предлагаемый план переноса

### Этап 1. Отделить gameplay core от браузера

Цель:

- чтобы игра могла жить без DOM-рендера

Что сделать:

- убрать из `GameCore` прямую зависимость от `document`
- свести `renderer`, `hud`, `audio` к интерфейсам-адаптерам
- перевести ключевые игровые действия на событийную модель

Результат:

- существующая игра продолжает работать в браузере
- но логика становится переносимой

### Этап 2. Зафиксировать формат данных

Что описать явно:

- сущность героя
- сущность врага
- состояние уровня
- состояние забега
- сюжетные флаги
- конфиги предметов, механик и scripted events

Лучше всего хранить это как явные data definitions, а не размазывать по коду.

### Этап 3. Собрать vertical slice в новом движке

Для Godot это будут:

- сцена уровня
- сцена героя
- сцена врага
- HUD
- одна scripted sequence

На этом этапе не нужно переносить все 3 этажа и весь UI.

### Этап 4. Переносить проект пакетами

Порядок:

1. базовое движение и камера
2. пошаговый цикл и AI
3. бой и предметы
4. HUD и меню
5. генерация уровня
6. кат-сцены и сюжетные триггеры
7. сохранения

### Этап 5. Заморозить старую браузерную ветку

После того как vertical slice в новом движке станет стабильным:

- старую версию оставить как reference branch
- всю новую разработку вести только в новом runtime

## Что я бы предложил сделать следующим коммитом

Если идти по прагматичному пути, следующий полезный шаг такой:

1. Вынести из [`GameCore`](../js/game/core/GameCore.js) прямые browser-specific зависимости.
2. Ввести интерфейс событий gameplay.
3. Подготовить слой адаптеров `browser renderer / browser hud / browser audio`.
4. После этого собрать `Godot spike` на одном уровне.

На момент сравнения shortlist выглядел так:

- основной кандидат: `Godot`

- альтернативный enterprise-вариант: `Unity`

Фактическое решение по проекту после этого сравнения: **переход в Phaser**.
Следующий практический шаг: **отделить gameplay core от DOM и собирать Phaser runtime поверх него**.
